1. AGENT BUILDER — compose, version, publish an agent
==============================================================================
WHAT IT IS
  The no-code UI for defining an agent: name, base instruction, model +
  temperature, output schema/key, and three kinds of attachments (tools,
  skills, sub-agents). Backs both admin- and developer-authored agents.

BUILT WITH
  Backend: FastAPI router (config_api/agents.py), SQLAlchemy 2.0 async ORM
  models (models/agents.py: Agent, AgentVersion, AgentTool, AgentSkill,
  AgentSubagent, AgentPublishRequest), Pydantic v2 schemas (schemas/agents.py)
  for request/response validation, Postgres (agent_forge schema).
  Frontend: React 19 + TypeScript page (pages/AgentBuilderPage.tsx) with
  sub-components (ToolAttachPanel, SkillAttachList/Picker, SubAgentAttachPanel,
  VersionHistory, VersionDiff, EffectivePromptPreview), @tanstack/react-query
  hooks (api/agents.ts) for all CRUD + attach/detach mutations with automatic
  cache invalidation, @dnd-kit for drag-reordering attached skills, `diff`
  (jsdiff) for word-level text diffing in the version-history view.

HOW IT WORKS
  - CRUD on `agents` writes to LIVE/draft tables only — nothing an end user
    (/chat) or external caller (/invoke) sees changes until PUBLISH.
  - Publish snapshots the full resolved config (instruction + skills +
    tools + sub-agents) as a frozen JSONB blob into a new `agent_versions`
    row, bumps `agents.current_version`, and invalidates the in-memory
    AgentCache for this agent AND every ancestor orchestrator (walking
    `agent_subagents` upward) — a published tree is fully materialized at
    build time, so a stale cached parent would otherwise keep serving an
    old child forever. Any script or admin action that republishes a
    specialist must walk and re-bump its ancestors too, or the orchestrator
    silently keeps serving the old child indefinitely — this bit twice
    during this build (see section 7's "lessons learned" and section 2).
  - Rollback restores the live draft to an older version's snapshot, then
    publishes THAT as a brand-new version — history is append-only, never
    overwritten.
  - The Version History dialog now does more than list versions: each row
    expands (chevron toggle) into a full, computed diff against the
    version immediately before it — instruction text word-diffed
    (additions highlighted green, removals struck through red), model
    config changed keys, and tools/skills/sub-agents added/removed shown
    as +/- badges. A version that changed nothing user-facing (e.g. a
    republish done purely to invalidate an ancestor's cache — see above)
    says so plainly instead of rendering an empty, confusing diff. An
    amber banner surfaces above the list whenever the LIVE draft has
    diverged from the last published snapshot ("unpublished changes since
    vN"), and the rollback confirmation dialog itself now shows a live
    "what this would change, right now" diff (current draft vs. the
    version being rolled back to) instead of only generic boilerplate
    text about the mechanics of rollback. All of this reuses one shared
    diff engine (components/agents/VersionDiff.tsx: `computeSnapshotDiff`/
    `hasSnapshotChanges`) so a version-vs-previous-version diff, a
    draft-vs-published diff, and a rollback preview are all the exact same
    comparison function pointed at different pairs of snapshots.
  - Circular sub-agent references are rejected on write (`_would_create_cycle`
    walks the graph from the proposed child to see if the parent is already
    reachable).
  - Every mutation writes a row to the hash-chained `config_audit_log` (see
    section 13) in the same request/transaction.
  - Publish itself branches by caller role — see section 16.
  - Frontend gate: the "Publish" button stays disabled until a Playground
    run has succeeded against the EXACT current configuration, fingerprinted
    (name+instruction+model+tools+skills+sub_agents) in lib/testGate.ts —
    you cannot publish something you haven't actually run.

2. AGENT RUNTIME / ORCHESTRATION ENGINE
==============================================================================
WHAT IT IS
  The compiler that turns Postgres rows into a real, runnable multi-agent
  tree, and the turn-execution machinery that actually drives a
  conversation through Google ADK + Gemini.

BUILT WITH
  Google ADK (google.adk.agents.Agent, google.adk.runners.Runner,
  google.adk.sessions — InMemorySessionService for Playground,
  DatabaseSessionService for /invoke and /chat), Gemini (gemini-2.5-flash
  default, per-agent configurable), backend/app/agent_runtime/ (builder.py,
  cache.py, schema_utils.py), backend/app/playground_api/router.py (the
  shared turn-execution engine every run surface funnels through).

HOW IT WORKS
  - Two build paths, both recursive: `_build_from_live_config` (reads
    draft tables, never cached — Playground) and `_build_from_snapshot`
    (reads the frozen `agent_versions.snapshot`, cached in AgentCache keyed
    by (agent_id, version) — /invoke and /chat).
  - ADK auto-grants every agent with sub_agents a `transfer_to_agent`
    function tool; routing between orchestrator and specialists is NOT
    code-level logic — it's the LLM reading the instruction text and
    deciding to call transfer_to_agent. `disallow_transfer_to_parent=False`
    on every built agent lets a specialist hand back to its orchestrator
    mid-conversation instead of being stuck once a session resumes it.
    Because routing is pure prose, not code, it is only as reliable as the
    instruction text: a routing bullet naming a sub-agent that was never
    actually attached fails the transfer outright (found live twice — see
    the "orchestration audit" note in section 7), and a generic instruction
    tacked onto the end of an agent's own numbered checklist can get
    silently ignored if that checklist reads as already-complete without
    it (see section 9's chart-tool rollout for a concrete case and fix).
  - `close_agent_toolsets()` (agent_runtime/builder.py) walks a built agent
    tree (itself + every sub_agent, recursively) and closes every
    `BaseToolset` it finds (e.g. an `McpToolset`, which owns a live MCP
    subprocess). This exists because of a real, live-diagnosed bug: a
    Playground run always builds a BRAND NEW agent tree (never cached),
    so every distinct MCP tool touched in that turn spawned a fresh
    subprocess — and nothing closed the previous turn's subprocesses
    afterward, since `McpToolset`'s own session pooling only helps when
    the SAME toolset instance is reused across calls, which a live/
    uncached build never does. Confirmed live: 124 orphaned
    `mcp_servers/*.py` subprocesses had accumulated over one session's
    testing before the fix (`playground_api/router.py::run_playground`
    now calls `close_agent_toolsets()` in a `finally` block after every
    turn). A cached (/invoke, /chat) build is deliberately NOT closed this
    way — its toolsets are meant to be reused across many requests for as
    long as that version stays cached.
  - `_execute_run`/`_stream_turn` (playground_api/router.py) consume ADK's
    Runner event stream, extracting tool calls, transfers, token usage, and
    final text — with reliability layers on the blocking path only
    (`_run_turn`): a same-session resample if the orchestrator hallucinates
    a direct tool call instead of transferring, and a session
    delete-and-recreate if a resumed session references state that no
    longer exists (stale config drift).
  - Both execution paths wrap themselves in OpenTelemetry spans (see
    section 14) and fire-and-forget-log an InvocationLog + one ToolCallLog
    row per tool call (logging_hooks.py) — never blocking the response.

3. TOOL REGISTRY — 7 tool types, zero stored code
==============================================================================
WHAT IT IS
  The library of reusable, attachable capabilities an agent can call.
  Every tool is a row of declarative JSONB config — never raw executable
  code — dispatched to a fixed Python class per type.

BUILT WITH
  backend/app/tool_registry/ — one file per type: http_tool.py, sql_tool.py,
  mcp_tool.py, retrieval_tool.py, image_gen_tool.py, nl2sql_tool.py
  (db_schema_tool + nl2sql_query_tool), factory.py (the type -> class
  dispatcher), serialize.py (JSON-safety for tool output). httpx for all
  outbound HTTP, SQLAlchemy `text()` with bound parameters for sql_tool/
  retrieval_tool (structure is fixed by config; only VALUES are
  LLM-controlled — no SQL injection surface), pgvector for retrieval_tool's
  knowledge-base search, MCP Python SDK for mcp_tool (see section 7).
  Frontend: components/tools/ (ToolForm, ToolCard, ToolTypeConfigFields —
  a type-specific config-field renderer, ParamSchemaBuilder for building
  the tool's input JSON Schema visually).

HOW IT WORKS
  config_api/tools.py is admin-only end to end (create/update/delete) —
  the one place in this platform where write access stays admin-exclusive
  even for the developer role, because these configs are a real security
  boundary (arbitrary URLs, SQL text, subprocess commands). A Tool row is
  reusable across many agents (agent_tools is a many-to-many join) — e.g.
  export_to_pdf is shared by all 5 market-intelligence-family analyst
  agents, and generate_chart_tool (section 9) the same way.

4. SKILLS LIBRARY
==============================================================================
WHAT IT IS
  Reusable instruction fragments (plus optional few-shot examples) that
  concatenate onto an agent's base_instruction, in a configurable order —
  a way to compose behavior without retyping the same paragraph into every
  agent that needs it.

BUILT WITH
  config_api/skills.py, models/skills.py, schemas/skills.py. Frontend:
  components/skills/ (SkillForm, SkillCard, FewShotEditor), api/skills.ts.

HOW IT WORKS
  `compose_instruction()` (agent_runtime/builder.py) appends each attached
  skill's instruction_text after the base instruction, in `attach_order` —
  the exact same function backs the live "Effective Prompt Preview" panel
  in the Agent Builder, so what you preview is what the model receives,
  byte for byte. Skills are pure text (no execution surface), so — unlike
  tools — the developer role has full create/update/delete access, not
  just read.

5. PLAYGROUND
==============================================================================
WHAT IT IS
  The "run this right now" testing surface for a DRAFT agent config,
  before anything is published.

BUILT WITH
  playground_api/router.py (`POST /playground/run`), InMemorySessionService
  (ADK) — sessions are cheap and throwaway by design, a backend restart
  should clear them. Frontend: pages/PlaygroundPage.tsx +
  components/playground/PlaygroundChat.tsx.

HOW IT WORKS
  Always rebuilds the agent tree fresh from the LIVE tables (never cached),
  so iterating on an unpublished edit is reflected on the very next run —
  and, since this build is never cached and nothing else will ever reuse
  it, every run's toolsets (MCP subprocesses included) are explicitly
  closed afterward via `close_agent_toolsets()` (section 2) rather than
  left to leak. Every successful run stamps a fingerprint (lib/testGate.ts)
  that the Agent Builder's Publish button checks before allowing a publish.

6. CHAT — the white-labeled end-user surface
==============================================================================
WHAT IT IS
  A separate, closed conversational UI (/chat) for real end users — always
  talks to exactly one published root agent (CHATBOT_AGENT_NAME =
  "market_intelligence" by default, or a caller-named orchestrator), never
  exposes the admin tooling.

BUILT WITH
  chat_api/router.py + chat_api/deps.py (require_chat_access), ADK
  DatabaseSessionService (Postgres-backed, survives a restart). Frontend:
  pages/ChatPage.tsx, components/chat/ (MessageRendering.tsx — markdown +
  structured quiz/flashcard rendering, ThinkingSteps.tsx — live progress
  UI), react-markdown + remark-gfm (renders a specialist's inline
  `![alt](url)` chart image — section 9 — the same way it renders any
  other markdown image, no special-casing needed).

HOW IT WORKS
  - `POST /chat/message` is blocking (full reliability layer via
    `_run_turn`); `POST /chat/message/stream` is newline-delimited-JSON
    streaming (`_stream_turn`) that taps ADK's own internal event stream
    live — transfer/tool_call_start/tool_call_end/done/error events — so
    the UI shows a live spinner-to-checkmark list instead of a blank wait
    during a multi-tool-call chain. Deliberately simpler than the blocking
    path: no stale-session/hallucination retry, just an "error" event.
  - Uses its OWN token (`af_user_token`, from `/user-login` or
    `/register`), separate from the admin-shell's `af_admin_token` — a
    developer/admin who reaches Chat via the admin sidebar has this
    bridged automatically at login time so it doesn't require a second
    sign-in (see section 16/20).
  - Self-service usage: `GET /chat/usage` (chat_user AND developer role)
    returns only the CALLING user's own invocation/token/cost stats,
    filtered by `invoked_by` — distinct from the admin-only
    `/dashboards/usage/*` endpoints that see everyone.

7. MARKET INTELLIGENCE AGENT FAMILY & MCP SERVERS
==============================================================================
WHAT IT IS
  The flagship, ready-to-use example of onboarding a real domain: an
  orchestrator (`market_intelligence`) plus 6 specialists — stocks, crypto,
  forex/metals, Indian mutual funds, company research, and slide-deck sales
  reporting (section 8) — backed by free/no-API-key public data plus one
  dedicated MySQL analytics database.

BUILT WITH
  MCP (Model Context Protocol) Python SDK — `mcp.server.fastmcp.FastMCP`.
  Each file in backend/mcp_servers/*.py (stocks, crypto, forex_metals,
  mutual_fund, weather, document_export, chart, slide_reporting) is a
  standalone MCP server spawned fresh, per tool call, as a stdio subprocess
  (StdioServerParameters) by tool_registry/mcp_tool.py — no shared process
  state, so an MCP server code edit takes effect on the very next call, no
  backend restart needed. Data sources: Yahoo Finance (stocks), CoinGecko
  (crypto), frankfurter.app + gold-api.com (forex/metals), mfapi.in/AMFI
  (mutual funds), Open-Meteo (weather) — all free, unauthenticated, no key.

HOW IT WORKS
  `market_intelligence` is a pure router (0 tools of its own, 6
  sub-agents) — the LLM reads its routing instructions and calls
  `transfer_to_agent`. Real statistical work happens in the MCP tools
  themselves (e.g. `predict_fund_trend` computes real annualized volatility
  + max drawdown from actual NAV history and projects forward using
  trailing CAGR — explicitly framed as extrapolation, never advice). Five
  of the six specialists (all but slide_reporting_agent, which already
  produces a full deck) also share `generate_chart_tool` (section 9) so a
  trailing-returns or comparison answer renders an inline chart, not just
  a table of numbers.

  ORCHESTRATION AUDIT (static, no extra LLM calls) — after growing
  "hallucinations in orchestration" reports, every real orchestrator's
  routing instruction text was cross-checked, line by line, against its
  ACTUAL attached sub-agents. Two agents were referenced in routing prose
  but never actually wired as a `sub_agent` at all — a `transfer_to_agent`
  call to either would simply fail: `fund_research_agent` (in StudyBuddy's
  root orchestrator) and, caught only while re-verifying that very fix,
  `example_agent` (same orchestrator) — both fixed by attaching them
  properly. `market_intelligence`'s own routing bullets were also
  restructured (a specialist's bullet had been appended after the "pick
  exactly one and transfer" rule instead of inside the main specialist
  list) and `stock_market_analyst` vs. `company_research_copilot`'s
  overlapping scope was given sharper, less ambiguous routing language.
  StudyBuddy's root orchestrator (school-study persona) had also picked up
  unrelated cross-wiring to `fund_analyst_agent` (adult financial
  disclaimers) and `weather_agent` — removed as a scope/tone mismatch.
  `backend/scripts/audit_fix_agents.py` carries out these fixes
  idempotently and is a reusable pattern for re-running this class of
  audit later.

8. SLIDE REPORTING AGENT (NL question -> SQL -> chart -> PPTX)
==============================================================================
WHAT IT IS
  A market-intelligence specialist that turns a natural-language sales/
  orders/revenue question into a downloadable PowerPoint deck — SQL
  generation, guarded execution, chart/slide planning, and rendering, each
  a separate chained MCP tool call. Targets its own dedicated MySQL
  database (`sales_analytics` — regions, sales_reps, categories, products,
  customers, orders, order_items, sales_targets), kept entirely separate
  from the app's own Postgres `agent_forge` schema.

BUILT WITH
  backend/mcp_servers/slide_reporting_server.py (4 MCP tools) plus shared
  sibling modules: `_schema_context.py` (a hand-written, concise grounding
  description of the MySQL schema fed into the NL->SQL prompt instead of
  raw DDL), `_db.py` (pooled MySQL access via PyMySQL + DBUtils'
  `PooledDB`), `_chart_utils.py` (shared with section 9's chart tool).
  `sqlglot` parses generated SQL into a real AST for validation — dialect-
  aware ("mysql"), never string/keyword matching. `google-genai` is called
  DIRECTLY from this standalone MCP subprocess (not as an ADK sub-agent
  call) for both NL->SQL generation and the insights/bullets summary,
  mirroring the same direct-Gemini pattern `tool_registry/image_gen_tool.py`
  already used elsewhere. `python-pptx` renders the deck; `matplotlib`
  (via `_chart_utils.py`) renders chart images embedded into slides.

HOW IT WORKS
  - Four tools, called in strict order by the specialist's own instructions
    (not enforced by any framework-level sequencing): `nl_to_sql_tool` ->
    `sql_execution_tool` -> `chart_planner_tool` -> `slide_builder_tool` —
    each tool's JSON output is read by the LLM and re-passed as the next
    tool's arguments, so the model itself is the pipe between steps.
  - SQL safety is layered, not just prompted: `sqlglot.parse()` rejects
    anything that isn't exactly one `SELECT`/`WITH...SELECT` statement,
    structurally (an `exp.Select` type check, not a regex); any
    INSERT/UPDATE/DELETE/DROP/ALTER/CREATE/TRUNCATE node anywhere in the
    parsed tree is rejected even if buried in a subquery; a LIMIT is
    appended via an AST edit (`Select.limit(...)`), never string
    concatenation, only when the query doesn't already have one; both
    `nl_to_sql_tool` (right after generation) and `sql_execution_tool`
    (independently, in case it's ever called with hand-written SQL)
    re-validate. A read-only MySQL user is the recommended (not yet
    enforced) defense-in-depth layer beyond this.
  - `sql_execution_tool` runs inside `SET SESSION MAX_EXECUTION_TIME=10000`
    to bound a runaway query, and reports whether truncation happened
    ONLY when it added the row cap itself — a user-requested "top 10"
    LIMIT is never confused with a safety truncation.
  - `chart_planner_tool` classifies the result shape with deterministic
    rules first (1 row -> KPI; one category + one metric under 15
    categories -> bar; a date/month column -> line; more categories or
    dimensions -> a paginated table), falling back to a single small
    Gemini call only when the rules genuinely can't decide — keeping the
    common case fast and free of an extra LLM round-trip. It always
    prepends a title slide and appends a short LLM-generated "key
    takeaways" bullets slide.
  - `slide_builder_tool` renders bar/line charts as matplotlib images (the
    same shared visual system as section 9, so a chart looks identical
    whether it lands in a slide deck or a chat bubble), tables paginated
    at 12 rows/slide with a "showing all N rows across K slides" note, a
    big-number KPI slide, and a plain "No data found for this query" slide
    for an empty result instead of crashing. On a SQL error, the
    specialist retries `nl_to_sql_tool` exactly once with the error as
    feedback before giving up with a clear failure message — never a
    silent hang or a fabricated answer.
  - Deck files land in `backend/generated_files/` (same directory and
    `/generated-files` static mount section 10's PDF/Excel export already
    uses) with an absolute `BACKEND_PUBLIC_URL` download link.
  - Registration: `backend/scripts/seed_slide_reporting_agent.py` creates
    the specialist + its 4 tools and attaches it to `market_intelligence`,
    loading the orchestrator's LIVE current instruction from the database
    (never a hardcoded constant) before appending one routing line, so any
    instruction edits made since seeding aren't clobbered — and, like
    section 9's rollout, walks and republishes every PUBLISHED ancestor
    orchestrator so no cached tree keeps serving a stale specialist list.

9. SHARED CHART GENERATION (generate_chart_tool)
==============================================================================
WHAT IT IS
  One shared MCP tool any market-intelligence specialist can call to turn
  a numeric series it already has — trailing returns, price/NAV history, a
  multi-fund/stock/coin comparison — into an inline chart image for its
  chat reply, without duplicating chart-rendering code per agent.

BUILT WITH
  backend/mcp_servers/chart_server.py (the tool itself) and
  backend/mcp_servers/_chart_utils.py — extracted out of section 8's
  slide-deck renderer specifically so both consumers (a PPTX slide and a
  chat-embedded PNG) share one implementation and therefore one visual
  system. matplotlib does the actual rendering; colors/marks follow the
  project's dataviz-skill palette: a fixed-order categorical hue set
  (never cycled) for 2+ series, a single sequential hue with no legend box
  for exactly one series (per the "a single series needs no legend" rule),
  recessive hairline gridlines, and auto-compact axis-tick formatting
  (1,284 / 12.9K / 4.2M) via a shared `MaxNLocator`+`FuncFormatter` pair
  chosen specifically so adjacent gridlines never round to the same
  displayed label.

HOW IT WORKS
  - `generate_chart_tool(chart_type, title, x_labels, series, y_label)`
    accepts one-or-more named series; a single series renders as a plain
    chart (no legend, sequential hue), two or more render as grouped bars
    or multiple lines with a legend and the fixed categorical palette —
    the exact same `render_bar_chart`/`render_line_chart` functions
    section 8 uses for slide decks.
  - Returns a READY-TO-PASTE markdown image snippet
    (`![title](BACKEND_PUBLIC_URL/generated-images/<uuid>.png)`), not just
    a bare URL — an absolute URL specifically, for the same reason as
    section 10's exports: the Vite dev server and the FastAPI backend run
    on different origins, so a relative link 404s against the wrong one.
    Returning the whole snippet means a specialist can't get the markdown
    syntax wrong.
  - Registered by `backend/scripts/seed_chart_tool.py` onto all 5
    non-slide-deck market-intelligence specialists, appending one shared
    instruction block rather than duplicating chart-calling prose five
    times. That instruction block had to be rewritten stronger than a
    first pass: `fund_analyst_agent`'s OWN numbered "for a full analysis,
    call these two tools" checklist doesn't mention charting at all, so a
    model following that checklist literally never reached a
    charting instruction appended after it — confirmed via a live
    invocation/tool-call-log trace showing a "complete analysis" question
    skip the chart tool entirely. The fix explicitly states it overrides
    any checklist above that "looks complete" without it.
  - Registering this tool onto five already-published, already-in-use
    agents is exactly the multi-ancestor cache-invalidation case section 1
    describes: `seed_chart_tool.py` walks and re-bumps every PUBLISHED
    ancestor of each changed specialist (found 3 for `fund_analyst_agent`
    alone: `market_intelligence`, StudyBuddy's `orchestrator`, and a third,
    archived orchestrator it also happened to be attached to) — and
    deliberately skips any non-published ancestor, since an early version
    of this same script briefly and accidentally un-archived one by
    republishing it just to bump its version for cache invalidation.

10. DOCUMENT EXPORT (PDF / Excel)
==============================================================================
WHAT IT IS
  Lets any market-intelligence specialist turn its OWN analysis into a
  downloadable file, in the same conversation, no separate "export agent"
  hand-off needed.

BUILT WITH
  reportlab (PDF — a best-effort markdown-to-PDF converter: headings,
  bold/italic/code, bullet lists, GFM tables), openpyxl (Excel — styled
  header row, auto-sized columns), served as static files via FastAPI's
  StaticFiles mount (/generated-files, /generated-images).

HOW IT WORKS
  `export_to_pdf`/`export_to_excel` are just two more MCP tools
  (document_export_server.py) attached to 5 different agents — pure local
  computation, no external API. Returns an ABSOLUTE URL via a
  BACKEND_PUBLIC_URL setting (a relative URL would resolve against the
  frontend's own origin in dev, on a different port, and 404) — the same
  convention section 8/9's newer file/image-producing tools follow.

11. MONITORING DASHBOARD
==============================================================================
WHAT IT IS
  Operational health: invocation volume, error rate, latency percentiles,
  broken down overall / per-agent / per-tool, over a configurable time
  window.

BUILT WITH
  dashboards_api/monitoring.py — raw SQL aggregation via SQLAlchemy's
  `func`/`case`/`within_group` (Postgres `percentile_cont` for real p50/
  p95/p99, not an approximation), no separate metrics store. Frontend:
  pages/MonitoringPage.tsx, StatTile components, Badge tone-coded health
  rows — no charting library here, just tables + tiles.

HOW IT WORKS
  Every query is workspace-scoped and windowed off `invocation_log.
  created_at`. Per-agent/per-tool rows are computed via a GROUP BY
  subquery outer-joined back onto `agents`/`tools` (so an agent/tool with
  ZERO calls in the window still shows up, coalesced to 0) and sorted
  worst-first (highest error rate, then highest p95 latency) so problems
  surface at the top. admin + viewer only (developer has no access to this
  admin-only dashboard family — see the permission matrix in section 16).

12. USAGE & COST DASHBOARD
==============================================================================
WHAT IT IS
  Cost/token accounting: summary, a daily timeseries chart, breakdowns by
  agent, by tool, and by user — plus the self-service "my own usage" view
  chat_user/developer accounts get (section 6).

BUILT WITH
  dashboards_api/usage.py (admin-only, sees everyone),
  observability/pricing.py (a hardcoded $/1M-token table per model prefix,
  explicitly presented as an ESTIMATE, never billing-accurate). Frontend:
  pages/UsagePage.tsx using recharts for the timeseries chart, with
  chartPalette.ts's `useIsDarkMode()`/`buildCategoricalScale()` for
  theme-correct, colorblind-safe categorical colors (tracks the app's
  actual selected theme via a MutationObserver on <html>'s class, not the
  OS media query — see bug #9 in the technical design document).

HOW IT WORKS
  `estimate_cost_usd(model, input_tokens, output_tokens)` is computed once
  per invocation at log-write time (logging_hooks.py) and stored on
  `invocation_log.estimated_cost_usd` — dashboards just SUM/AVG that
  column, they don't re-derive pricing per query. The by-user breakdown
  joins `invoked_by` (a free-form string: a real user UUID for /chat
  callers, or a label like "playground-user"/"external-caller" for
  admin-facing surfaces) back to `users` in PYTHON, not SQL, specifically
  to avoid a cast error on values that were never a UUID to begin with.

13. AUDIT LOG — tamper-evident configuration + invocation history
==============================================================================
WHAT IT IS
  Two searchable, exportable logs: every config mutation (agent/tool/skill
  create/update/publish/archive/delete) and every agent invocation
  (transcript, latency, cost, status) — the config side is cryptographically
  tamper-evident.

BUILT WITH
  dashboards_api/audit.py, audit_hash.py (a dependency-free SHA-256
  hash-chain function shared between the app and an Alembic backfill
  migration), models/logs.py (ConfigAuditLog, InvocationLog). Frontend:
  pages/AuditPage.tsx (tabbed invocations/config-changes views, CSV/JSON
  export, a detail modal per row).

HOW IT WORKS
  Each `config_audit_log` row's `row_hash` = SHA-256(prev_hash + entity +
  action + actor + diff + timestamp) — a hash chain, so alt­ering or
  deleting any past row breaks every row_hash after it.
  `GET /dashboards/audit/verify-chain` recomputes the entire chain from
  scratch and confirms it matches what's stored — proof the trail hasn't
  been edited, including by someone with direct DB access. Writes are
  serialized with `SELECT ... FOR UPDATE` on the last row so two
  concurrent writers can't compute the same seq/prev_hash and silently
  fork the chain. `write_audit_log()` is awaited INLINE in the same
  request/transaction as the mutation it's recording (correctness over
  latency here) — unlike invocation logging, which is fire-and-forget.

14. DEBUG CONSOLE — multi-agent trace/waterfall viewer
==============================================================================
WHAT IT IS
  Lets you trace exactly what happened across one multi-agent turn: which
  specialist handled it, every tool call it made, and how long each step
  took — a real waterfall, not just an aggregate log line.

BUILT WITH
  OpenTelemetry (opentelemetry-api/sdk, -instrumentation-fastapi/-httpx,
  -exporter-otlp-proto-grpc) — app/observability/tracing.py sets up a
  TracerProvider + BatchSpanProcessor exporting OTLP/gRPC, OFF by default
  (OTEL_ENABLED=false). Jaeger (jaegertracing/all-in-one via
  docker-compose.yml at the repo root) as the reference trace backend —
  but nothing is Jaeger-specific except the optional Query-API read-back;
  point OTEL_EXPORTER_OTLP_ENDPOINT at Tempo/Langfuse/Honeycomb/Datadog/
  anything OTLP-compatible instead and spans just show up there too.
  Backend: app/debug_api/router.py, schemas/debug.py. Frontend:
  pages/DebugConsolePage.tsx (trace list + a hand-built CSS waterfall —
  proportionally-sized/offset <div> bars, no charting library), api/debug.ts.

HOW IT WORKS
  Two operating modes, same UI either way:
  - "reconstructed" (always available, zero extra infra): one root span
    from the InvocationLog row, one child span per ToolCallLog row, laid
    out sequentially via ToolCallLog.call_index (NOT created_at — every
    one invocation's tool-call rows land in a single fire-and-forget batch
    write, so creation timestamps can't recover true call order) and
    cumulative latency_ms — approximate timing, but real per-tool-call
    agent attribution (agent_name column) and real latency.
  - "jaeger" (when OTEL_ENABLED=true and the trace landed there): fetches
    the real span tree from Jaeger's Query API, real nanosecond-precision
    timing, with an "Open in Jaeger" deep link for the full Jaeger UI.
  `playground_api/router.py`'s `_execute_run`/`_stream_turn` open one root
  span ("agent.invocation") per turn and one child span ("tool.<name>")
  per ADK tool call, tagged with whichever agent actually made that call
  (from ADK's `event.author`) — this is what closed the previously-known
  gap where a chat turn's tool calls couldn't be attributed to the
  specialist that made them.
  Access is role- and ownership-scoped: admin/viewer see the whole
  workspace; developer sees only traces for agents they created or
  invocations they personally triggered.

15. AUTHENTICATION & ROLES
==============================================================================
WHAT IT IS
  Two credential types, four roles (admin, viewer, chat_user, developer),
  gating every router.

BUILT WITH
  auth_api/router.py, auth_users.py (dependency-free PBKDF2 password
  hashing + a hand-rolled HMAC-signed session token — deliberately not
  pulling in passlib/PyJWT for this scope), principal.py (the single
  `Principal` dataclass + `require_role(...)`/`get_current_principal`
  dependency every router hangs off), models/users.py.

HOW IT WORKS
  - AGENT_FORGE_API_TOKEN: a static break-glass Bearer token, always
    resolves to role=admin, workspace=DEFAULT_WORKSPACE_ID — no DB lookup.
  - Per-user tokens: `issue_user_token(user_id)` at `/auth/login`, a
    base64 JSON payload {user_id, exp} + an HMAC-SHA256 signature keyed on
    USER_TOKEN_SECRET, verified on every request via
    `verify_user_token()`. 30-day TTL.
  - Public self-registration (`POST /auth/register`) always creates a
    PENDING account — role limited to SELF_SERVE_ROLES (chat_user,
    developer); admin/viewer are only creatable directly by an existing
    admin (`POST /auth/users`, pre-approved, no pending queue — an admin
    is vouching for them).
  - `require_role("admin", "developer")` etc. is a FastAPI dependency
    factory — every router declares exactly which roles reach it; access
    control lives in ONE place (principal.py), never re-implemented
    per-route.

16. DEVELOPER ROLE & PUBLISH-APPROVAL WORKFLOW
==============================================================================
WHAT IT IS
  A 4th role: enough access to build/test/onboard agents end to end
  without being fully trusted to ship them straight to real users.

BUILT WITH
  Same stack as section 1/15, plus a new table (models/agents.py:
  AgentPublishRequest) and router (config_api/publish_requests.py).

HOW IT WORKS — TWO INDEPENDENT GATES
  1. Account approval (existing pending/approved/rejected on `users`) —
     a one-time "let this person in the door" decision, same flow as
     chat_user.
  2. Per-publish approval (new) — a developer's `POST /agents/{id}/publish`
     NEVER goes live directly. It freezes the current config into an
     `AgentPublishRequest` row (status=pending); an admin reviews the
     EXACT frozen snapshot at `GET /agents/publish-requests` and
     approves (publishes exactly that snapshot via the same `_publish_now`
     helper an admin's own direct publish uses) or rejects (with an
     optional note the developer sees at `/agents/publish-requests/mine`).
     At most one pending request per agent at a time (409 on a second
     attempt) — no way to spam the queue.
  Ownership scoping: a developer can only create new agents (created_by is
  server-set to their own identity, never trusted from the request body)
  and can only modify (update/archive/attach/publish) agents THEY created
  — `_require_can_modify()` in config_api/agents.py. Tools stay
  admin-only to create (real security boundary); Skills are open to
  developer CRUD (inert text, no execution surface). Rollback stays
  admin-only even for a developer's own agent, deliberately — it
  republishes immediately with no review step, which would bypass gate 2
  entirely.
  Chat access: developer is in chat_api/deps.py's _CHAT_ROLES — they can
  dogfood agents they're building in the real chat surface.
  Frontend: the admin shell (Layout.tsx) now accepts a per-user
  email/password login (not just the static token) via LoginPage.tsx,
  storing role in localStorage (lib/auth.ts's getStoredRole/setStoredRole)
  to drive a role-aware sidebar — a developer sees Agents/Tools(read)/
  Skills/Chat/Debug Console/"My publish requests", never Users or the
  admin-only dashboards. New PublishRequestsPage.tsx serves both the
  admin's full review queue and (via a `mine` prop) the developer's
  read-only view of their own requests' outcomes.

18. RATE LIMITING
==============================================================================
WHAT IT IS
  Per-principal sliding-window limiting on the request-triggering,
  cost-incurring routes (/chat/message, /playground/run, /invoke), plus a
  tighter IP-keyed limit on the pre-auth /auth/register and /auth/login.

BUILT WITH
  rate_limit.py — a plain in-process `dict[str, deque[float]]`, no Redis/
  external store (explicitly noted as a single-instance-only design; a
  multi-instance deployment would need a shared store instead).

HOW IT WORKS
  `rate_limit_principal`: 20 requests/60s, keyed by user_id (or
  `static:<role>` for the break-glass token) — every named account gets
  its own budget. `rate_limit_by_ip`: 10 requests/60s keyed by client IP,
  guarding registration/login specifically against brute-force/spam
  signups before any Principal even exists to key on.

19. THEME SYSTEM
==============================================================================
WHAT IT IS
  Six selectable themes (Light, Dark, Midnight, Sunset, Rose, Forest) plus
  System, via a ThemeToggle dropdown in the admin sidebar and /chat.

BUILT WITH
  Tailwind CSS v4 `@theme` custom properties + `@custom-variant dark`,
  React context (lib/theme.tsx — ThemeProvider/useTheme), a pre-paint
  inline script in index.html (avoids a flash of the wrong theme before
  React mounts), localStorage persistence.

HOW IT WORKS
  Each theme is a `:root[data-theme='<name>']` CSS block overriding
  --color-brand-*/--color-accent-*/shadow variables + a body background
  gradient — zero per-component rewrites. Dark-family themes (Dark,
  Midnight, Forest) also get a `.dark` class on <html>, which every
  existing Tailwind `dark:` utility already responds to. `color-scheme` is
  tied to the SAME `.dark` class (`:root.dark { color-scheme: dark }`)
  rather than left as the browser's OS-tracking default — otherwise native
  form-control text color (inputs/selects/textareas) would follow the
  OS's light/dark preference instead of the app's own theme, producing
  black-on-black text under a dark theme while the OS was still light
  (documented as bug #14 in the technical design document). Chart colors
  (chartPalette.ts, and the mcp_servers-side `_chart_utils.py` palette used
  by sections 8/9) independently track the same kind of light/dark surface
  for the identical reason.

21. DATABASE & SCHEMA MANAGEMENT
==============================================================================
WHAT IT IS
  The system-of-record database is Postgres, one dedicated schema
  (`agent_forge`) inside the shared `studybuddy` database — plus a SECOND,
  fully independent database engine (MySQL, `sales_analytics`) that exists
  purely as a real-world dataset for section 8's slide-reporting agent to
  query; the two are never joined and share no code path.

BUILT WITH
  PostgreSQL, SQLAlchemy 2.0 (fully async, asyncpg driver), Alembic
  (versioned migrations, one file per schema change — every table/column
  added this build has its own migration, never a hand-edited-in-place
  one), pgvector extension (retrieval_tool's embedding search). Separately:
  MySQL 8.x for `sales_analytics` (regions/sales_reps/categories/products/
  customers/orders/order_items/sales_targets — a deliberately "medium
  complex" schema: multi-level joins, an enum status field, a
  self-referencing category tree, discounts, targets-vs-actuals), reached
  only via PyMySQL + DBUtils' `PooledDB` from `mcp_servers/_db.py`, never
  through SQLAlchemy/asyncpg — a completely separate connection stack on
  purpose, so the two databases can't accidentally cross-contaminate
  connection pooling or transactions.

HOW IT WORKS
  Every Postgres table carries a `workspace_id` for multi-tenancy (enforced
  in query filters everywhere; not yet a hard DB-level boundary — a known,
  explicitly-documented limitation). Seed scripts
  (backend/scripts/seed_*.py) are idempotent with a `--reset` flag, used
  to stand up the StudyBuddy, Market Intelligence, and Slide Reporting
  agent families from scratch, and to attach shared tools (chart
  generation) onto already-live agents without disturbing anything else
  about them.
  The MySQL side has its own env-var namespace (`SALES_DB_HOST/PORT/USER/
  PASSWORD/NAME`) loaded directly from `backend/.env` by the MCP
  subprocess itself via `python-dotenv` — not through `app/config.py`'s
  Settings object, since a standalone MCP server doesn't share the FastAPI
  process's config at all (no other `mcp_servers/*.py` file does either).

22. LOCAL DEV / DEPLOYMENT INFRA
==============================================================================
WHAT IT IS
  What it takes to actually run this: the backend process, the frontend
  dev server, and the optional tracing backend.

BUILT WITH
  uvicorn (ASGI server, `--reload` for dev — noted as unreliable
  mid-session in practice; a manual kill+restart is the trustworthy way to
  guarantee new backend code is live), Vite dev server (frontend), Docker
  Compose (docker-compose.yml at the repo root — a single `jaeger`
  service, `jaegertracing/all-in-one`, OTLP receiver on :4317 + UI/Query
  API on :16686), python-dotenv/pydantic-settings for config
  (backend/.env, frontend/.env — VITE_API_BASE_URL must match whatever
  port the backend is actually listening on; a mismatch here has caused
  real "why isn't my change showing up" confusion more than once).
  Restart discipline matters more than it looks: a DB-only change (a new
  agent row, an updated instruction, a version bump) takes effect on the
  very next request with no restart, because AgentCache is keyed by
  (agent_id, version) and a version bump is a guaranteed cache miss — but
  a CODE change (e.g. section 2's `close_agent_toolsets` fix) is only
  picked up by a process that reloads that code, so an already-running
  uvicorn needs an actual restart before a backend code fix takes effect.
