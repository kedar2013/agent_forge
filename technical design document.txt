1. OVERVIEW & PURPOSE
==============================================================================
Agent Forge lets someone compose a working AI agent — instructions, tools,
reusable prompt fragments ("skills"), and sub-agents — entirely through a
web UI, with no code required for the composition itself. The agent is
stored as rows in Postgres, "compiled" into a real google.adk.agents.Agent
object at request time, and run against Gemini through ADK's Runner.

Two independent products are built on the same backend:
  - The ADMIN UI — build, test, publish, and monitor agents (/,  /agents,
    /tools, /skills, /monitoring, /usage, /audit, /users).
  - The CHAT UI (/chat) — a white-labeled conversational surface backed by
    exactly one published root agent (today: "market_intelligence"), used
    by real end users who never see the admin tooling.

The flagship capability built on this platform is a Market Intelligence
agent family: real-time stocks/ETFs/indices, cryptocurrency, forex &
precious metals, deep Indian mutual fund analysis (including statistical
trend/volatility predictions and SIP projections), and comprehensive
company research — all from free, no-API-key public data sources, with a
hard rule threaded through every agent: real computed data or an honest "I
don't have that," never a fabricated figure.

5. DATABASE SCHEMA (Postgres, schema "agent_forge")
==============================================================================
workspaces          multi-tenancy root (DEFAULT_WORKSPACE_ID today; the
                    columns exist everywhere but enforcement is partial).
users               email/password (hashed), role, workspace_id, status
                    (pending/approved).
agents               name, description, base_instruction, model_config
                    (JSONB: model + temperature), output_schema,
                    output_key, status (draft/published/archived),
                    current_version, workspace_id.
agent_versions        frozen JSONB snapshot per publish (name, instruction,
                    model_config, tools[], skills[], sub_agents[]) — this
                    is what /invoke and /chat actually run, NOT the live
                    draft tables.
agent_tools            many-to-many, agent <-> tool. A single Tool row can
                    be attached to MULTIPLE agents — e.g. export_to_pdf is
                    shared across 5 different agents.
agent_skills            many-to-many with attach_order (instruction
                    fragments concatenate onto base_instruction in order).
agent_subagents          many-to-many, parent_agent_id -> child_agent_id,
                    cycle-checked on write (_would_create_cycle walks the
                    graph from child to see if parent is reachable).
tools                  name, tool_type, config (JSONB), input_schema (JSON
                    Schema), workspace_id.
skills                 name, instruction_text, tags[], few_shot_examples
                    (JSONB), workspace_id.
invocation_log           one row per agent run: status, latency, tokens,
                    estimated_cost_usd, invoked_by (a user id or a caller
                    label like "playground-user"), trace_id, transcript
                    (message + response_text).
tool_call_log             one row per tool call within an invocation
                    (tool_id — currently only populated by the admin
                    dashboards' own instrumentation, NOT by _run_turn/
                    _stream_turn, which log tool call COUNT but not
                    per-call tool_id/name durably — see section 15).
config_audit_log            append-only, hash-chained: seq, prev_hash,
                    row_hash = SHA-256(prev_hash + entity/action/actor/
                    diff/timestamp). Tamper-evident — altering or deleting
                    a past row breaks every row_hash after it, and
                    /dashboards/audit/verify-chain proves it.

7. AGENT ORCHESTRATION — DEEP DIVE
==============================================================================

7.1 CORE CONCEPTS
------------------------------------------------------------------------------
- AGENT: a row in `agents` — a name, an instruction, a model, and three
  kinds of attachments: TOOLS (callable capabilities), SKILLS (instruction
  fragments concatenated onto the base instruction, in order), and
  SUB-AGENTS (other agents it can hand a conversation off to).
- ORCHESTRATOR vs SPECIALIST is not a schema distinction — it's a pattern.
  An "orchestrator" is simply an agent with sub_agents and (by convention)
  zero tools of its own, instructed to always transfer rather than answer.
  A "specialist" is a leaf: it has tools, no sub_agents, and actually does
  the work. Nothing in the data model enforces this split; it's discipline
  in how you write the instruction and what you attach.

7.2 HOW AN AGENT IS "COMPILED" (agent_runtime/builder.py)
------------------------------------------------------------------------------
Two build paths, both recursive (a sub-agent is built by recursively
building ITS sub-agents, tools, and skills):

  _build_from_live_config(db, agent_id)
    Reads the LIVE/draft tables directly. Never cached — every call
    rebuilds from whatever is currently saved, even unpublished changes.
    Used exclusively by /playground/run, so iterating on a draft agent
    always reflects your latest edit.

  _build_from_snapshot(db, agent_id, version)
    Reads the FROZEN `agent_versions.snapshot` JSONB for that specific
    version. Cached in AgentCache keyed by (agent_id, version). Used by
    /agents/{id}/invoke and /chat — the stable, published surface.

Both paths end by constructing a real `google.adk.agents.Agent`:
    AdkAgent(
        name=_safe_agent_name(name),      # sanitized to a valid identifier
        instruction=compose_instruction(base_instruction, skills),
        tools=[build_tool(t) for t in tools_rows],
        sub_agents=[... built recursively ...],
        output_schema=..., output_key=...,
        disallow_transfer_to_parent=False,   # see 7.5 — this was a bug fix
    )
Sub-agents are ALWAYS built fresh, never pulled from AgentCache even when
recursing — ADK mutates a `parent_agent` reference onto a child the moment
it's assigned into `sub_agents=[...]`, so a cached (already-parented)
instance can't safely be reused as anyone's child a second time.

7.3 THE TRANSFER MECHANISM
------------------------------------------------------------------------------
ADK automatically gives every agent with sub_agents a `transfer_to_agent`
function tool — the model calls it with the target agent's name, and ADK
switches execution to that agent for the rest of the turn (and, per 7.4,
often for future turns too). This is how "You are the Market Intelligence
orchestrator... transfer to the right specialist" instructions actually
take effect: the LLM reads the routing rules in its instruction text and
picks a transfer_to_agent call accordingly. There's no code-level routing
logic — the routing IS the instruction text, interpreted by the model.

7.4 SESSION RESUMPTION — WHY A CONVERSATION "STICKS" TO A SPECIALIST
------------------------------------------------------------------------------
ADK does not re-run the root agent on every turn. Runner._find_agent_to_run
walks the session's event history backwards looking for the most recent
non-user event's author:
  - If that author IS the root agent passed to Runner(agent=...) this
    call, resume the root.
  - Else if that author is found as a (possibly transitive) sub-agent of
    the CURRENT root and is "transferable" (see 7.5), resume THAT agent
    directly — skipping the root/routing step entirely.
  - Else (author not found in the current tree at all — e.g. a stale
    session from before an agent was retired) keep walking backward
    through older events looking for one that IS resolvable, and only
    fall back to the root agent if nothing in the whole history matches.

Practical effect: once a conversation transfers to a specialist, EVERY
subsequent message in that session_id goes straight to that specialist —
not back through the orchestrator — until something forces a change. This
is what "session_id" continuity buys you (no re-routing overhead turn
after turn), and it's also the mechanism behind two real bugs fixed this
session (7.5, 14).

7.5 disallow_transfer_to_parent — THE SINGLE MOST IMPORTANT FIX
------------------------------------------------------------------------------
Every built agent previously had `disallow_transfer_to_parent=True`
hardcoded, with the rationale "a specialist answers and stops rather than
looping back through its orchestrator." Combined with 7.4's stickiness,
this meant: once a session resumed a specialist, that specialist could
NEVER hand back to its orchestrator — even when asked something completely
outside its domain. A user who asked about a mutual fund, then asked about
Bitcoin in the SAME conversation, would get "I can only analyze Indian
mutual funds" forever, because the fund specialist had no way to say "not
my job, here — orchestrator, you take it."

Fixed by flipping this to `disallow_transfer_to_parent=False` for every
built agent. Verified directly: pin a session to fund_analyst_agent, then
ask about Bitcoin in the same session — it now correctly transfers back up
to market_intelligence and across to crypto_analyst, instead of staying
stuck.

7.6 THE ORCHESTRATOR HALLUCINATION EDGE CASE
------------------------------------------------------------------------------
market_intelligence has ZERO tools of its own by design (pure router — see
7.1). Occasionally Gemini hallucinates a direct call to a SPECIALIST's tool
(e.g. search_stocks) from the ORCHESTRATOR's own turn, instead of calling
transfer_to_agent first. ADK has no graceful fallback for an unresolvable
function name and raises hard:
    ValueError: Tool 'search_stocks' not found. Available tools: transfer_to_agent
`_run_turn` now catches this specific error shape ("not found" + "Available
tools" both present) and resamples the SAME turn once on the SAME session
(not a session wipe — that would cost potentially long, otherwise-healthy
conversation history just to fix one bad model sample). This is a
DIFFERENT recovery than 7.7's stale-session case, and deliberately so.

7.7 STALE-SESSION SELF-HEAL
------------------------------------------------------------------------------
A session can resume an agent that predates the CURRENT tree's config —
e.g. it last talked to an agent whose instruction referenced `{language}`
via ADK's `{state_key}` templating, but the session never had `language`
in its state (that agent's set of expected state keys changed, or the
session originated under a different chatbot root entirely). This raises
`KeyError: Context variable not found: \`language\`.` `_run_turn` catches
this and, UNLIKE 7.6, deletes and recreates the session before replaying —
here the history genuinely IS incompatible, so a same-session retry would
just fail identically.

7.8 PUBLISHING, VERSIONING, AND CASCADING CACHE INVALIDATION
------------------------------------------------------------------------------
Editing an agent's instruction/tools/sub_agents only changes the LIVE
tables — /chat and /invoke are unaffected until you PUBLISH, which:
  1. Snapshots the agent's current full config into a new `agent_versions`
     row (or overwrites version 1 in place if status is still "draft").
  2. Bumps `agents.current_version`, sets status="published".
  3. Invalidates AgentCache for this agent_id — AND, critically, for every
     ANCESTOR orchestrator too (walking agent_subagents upward via
     _find_ancestor_ids). This second part was a real bug: a published
     agent's build is a fully-materialized tree with children baked in at
     build time, keyed only on the PARENT's own version number. Publishing
     a child alone left any already-cached PARENT silently serving the old
     child forever, until the parent was ALSO republished. Fixed once,
     protects every future agent family, not just this one.
Publish is gated in the UI: a "Publish" button stays disabled until a
Playground run has succeeded against this EXACT configuration (name +
instruction + model + tools + skills + sub_agents fingerprinted via
lib/testGate.ts) — you cannot publish something you haven't actually run.

7.9 WORKED EXAMPLE — FULL REQUEST LIFECYCLE
------------------------------------------------------------------------------
User (via /chat) asks: "I want to start an SIP of 10,000 rupees a month.
Based on historical data and current trends, which sectors should I
target?"

  1. chat_api.send_chat_message resolves agent_row = the published
     "market_intelligence" agent for this principal's workspace.
  2. get_or_build_agent(db, agent_id, version=current_version) — cache hit
     (already built from an earlier request) or a fresh recursive build.
  3. _run_turn creates/resumes an ADK session (DatabaseSessionService,
     Postgres-backed) and calls Runner.run_async with the user message.
  4. market_intelligence's Gemini call decides this is a mutual-fund
     question and emits transfer_to_agent(agent_name="fund_analyst_agent").
  5. Control passes to fund_analyst_agent. Its instruction (see section 8)
     tells it: for a broad sector view, call scan_sector_trends_mcp.
  6. That MCP tool call spawns mutual_fund_server.py as a stdio subprocess,
     which fetches real NAV history for 10 representative sector funds
     from mfapi.in, computes real 3M/6M/1Y/3Y returns, and returns a
     ranked comparison — with an explicit caveat that this is historical
     data, not a recommendation.
  7. fund_analyst_agent's model turn formats that into markdown (a GFM
     table, per its formatting instruction) and returns final text.
  8. _run_turn collects the trace, fire-and-forget logs an InvocationLog
     row, and returns PlaygroundRunResponse to the client.
  9. (If the client used /chat/message/stream instead — see section 10.2 —
     steps 4-7 would have streamed live "transfer" and "tool_call_start/
     end" events to the UI as they happened, instead of the client seeing
     nothing until step 8.)

10. FEATURE DEEP-DIVES
==============================================================================
10.1 DOCUMENT EXPORT (PDF / EXCEL)
------------------------------------------------------------------------------
Any of the 5 market-intelligence specialists can turn its OWN analysis
into a downloadable file, in the same conversation, without a hand-off to
a separate "export agent" that wouldn't have the content:
  export_to_pdf(title, markdown_content) — a best-effort (not full
    CommonMark) markdown-to-reportlab converter handling headings, bold/
    italic/code spans, bullet lists, and GFM tables — the same markdown
    subset these agents already write in normal replies.
  export_to_excel(title, sheet_name, rows) — rows as a list of
    column-name -> value dicts; openpyxl renders a styled header row and
    auto-sized columns.
Files land in backend/generated_files/, mounted at /generated-files (same
pattern as the pre-existing image-generation tool's generated_images/).
Returns an ABSOLUTE URL via a BACKEND_PUBLIC_URL setting — a relative URL
would resolve against the FRONTEND's origin in dev (different port) and
404; this was caught and fixed before shipping.

10.2 LIVE STREAMING / "CHAIN OF THOUGHT"
------------------------------------------------------------------------------
Multi-tool-call chains (e.g. resolve a ticker, then analyze performance,
then export a PDF) can genuinely take 15-30+ seconds. POST
/chat/message/stream (newline-delimited JSON) taps into the SAME event
stream ADK's Runner already produces internally — previously fully
consumed and discarded until a final response — and yields each one live:
  {"type": "transfer", "to": "..."}
  {"type": "tool_call_start", "name": "...", "input": {...}}
  {"type": "tool_call_end", "name": "..."}
  {"type": "done", "response_text": "...", "tool_calls": [...], ...}
  {"type": "error", "message": "..."}
The chat UI shows a live spinner-to-checkmark list while pending, which
collapses into a small "Thought process (N steps)" toggle above the final
answer once it lands. Deliberately simpler than _run_turn: no stale-
session/hallucination retry here (that reliability layer stays on the
blocking endpoints) — an error mid-stream just ends the stream with an
"error" event.

10.3 SELF-SERVICE USAGE
------------------------------------------------------------------------------
GET /chat/usage (chat_user role, not just admin) returns the CALLING
user's own invocation count, token usage, cost, error count, and a daily
breakdown — filtered by `invoked_by`, which is the real user's id for
chat-originated calls. Distinct from the admin-only /dashboards/usage/*
endpoints, which see everyone's usage.

10.4 MUTUAL FUND DEEP ANALYSIS & PREDICTIONS
------------------------------------------------------------------------------
predict_fund_trend computes real annualized volatility and maximum
drawdown from the fund's actual monthly NAV series (last up to 36 months),
then projects NAV 6/12 months forward using the fund's own trailing CAGR
as a central estimate with a rough +/- band derived from that volatility.
Framed explicitly and repeatedly (in the tool's own output AND in every
consuming agent's instruction) as a mathematical extrapolation of the
past, NOT a forecast or investment advice — consistent with the
"data/analysis, never advice" disclaimer on every market-intelligence
agent. A real bug was caught and fixed here before shipping: the initial
implementation picked the WRONG historical NAV to anchor the CAGR
calculation (walking the date list in the wrong direction), producing a
nonsensical 162% "3-year CAGR" instead of the correct ~18% — caught by
cross-checking against the already-verified analyze_fund_performance
output for the same fund before ever wiring it into an agent.

13. SECURITY & AUTH MODEL
==============================================================================
Two credential types, both Bearer tokens:
  - AGENT_FORGE_API_TOKEN — a static admin break-glass token, always
    resolves to role=admin, workspace=DEFAULT_WORKSPACE_ID.

14. ARCHITECTURAL BUGS FOUND & FIXED THIS BUILD
==============================================================================
 1. disallow_transfer_to_parent hardcoded True — specialists could never
    hand back to their orchestrator once a session resumed them (7.5).
 2. Publishing a sub-agent didn't invalidate its orchestrator's cache —
    silent stale-child serving until the parent was also republished (7.8).
 3. Orchestrator tool-call hallucination — a pure-router agent with zero
    tools occasionally called a specialist's tool directly, raising a hard
    ValueError instead of transferring (7.6).
 4. Stale-session "Context variable not found" crashes — a session
    resuming an agent whose expected state keys had since changed (7.7).
 5. Empty-response turns — Gemini occasionally finishes a multi-hop
    transfer+tool-call turn with no text part; now falls back to the last
    tool result in plain language instead of a blank bubble.
 6. Wrong backend process being edited/restarted for half a debugging
    session — frontend's VITE_API_BASE_URL pointed to port 8130, not the
    8000 test instance being iterated on. Always confirm the actual port
    in frontend/.env before assuming a restart took effect.
 7. Tool/Skill cards had no edit affordance at all (only delete) — a
    genuine "no-code platform can't actually edit config" gap.
 8. Recharts axis-tick text rendered in a hardcoded near-black color
    invisible against the Midnight theme — SVG <text> doesn't inherit CSS
    color without an explicit stroke="currentColor" + text-color class
    treatment; fixed to match the one existing chart that already did this
    correctly.
 9. Chart dark-mode color detection read the OS prefers-color-scheme media
    query instead of the app's actual selected theme — wrong colors when
    OS and in-app theme selection disagreed. Fixed with a MutationObserver
    on <html>'s class list.
10. A relative generated-file URL would 404 from the frontend's origin in
    dev (different port than the backend) — fixed to return an absolute
    URL via a BACKEND_PUBLIC_URL setting before it ever shipped.
11. company_research_copilot (built via the UI onboarding walkthrough) was
    missing 4 of its instructed tools (get_quarterly_earnings,
    export_to_pdf, export_to_excel, plus an earlier gap in 3 company-
    research tools) — a reminder that attaching tools one at a time via
    the UI picker is easy to under-complete for an agent with many tools;
    worth a "does every tool your instruction mentions actually appear in
    your attached list" sanity check after building any new agent.
12. predict_fund_trend's CAGR anchor-date lookup walked the historical
    series in the wrong direction, producing a wildly wrong figure —
    caught by cross-checking against an already-verified figure before
    the tool was ever wired into a live agent.
13. GEMINI_API_KEY went blank in backend/.env at some point outside this
    session's own changes — masked for a while because the THEN-running
    backend process had already cached a valid Settings object in memory
    (lru_cache on get_settings()) from before the key was cleared, so it
    kept working until the next restart re-read the now-empty .env.
14. Unstyled form controls (input/select/textarea text color) rendered
    black-on-black under a dark-family theme (Dark/Midnight/Forest)
    whenever the OS itself was still in light mode. Root cause: `:root {
    color-scheme: light dark }` lets the browser pick native form-control
    text color from prefers-color-scheme (the OS setting) — completely
    independent of this app's own .dark class — so our Tailwind
    dark:bg-slate-800 correctly darkened the input's background while the
    browser's own native text color stayed the light-mode default (near-
    black). The SAME class of bug as #9 above (OS vs in-app theme
    disagreement), just hitting native form-control rendering instead of
    chart colors. Fixed the same way: keyed color-scheme off the existing
    `.dark` class (:root.dark { color-scheme: dark }) instead of the OS
    media query. Also swept the codebase and added an explicit
    dark:text-slate-100 to every input/select/textarea that had been
    relying on inherited/native text color instead of stating it — defense
    in depth so correctness doesn't depend on browser color-scheme
    behavior at all.

15. KNOWN LIMITATIONS / DELIBERATELY NOT BUILT
==============================================================================
- True single-turn cross-domain synthesis (e.g. "compare gold to the S&P
  500" as ONE combined answer) doesn't exist — a specialist answers its
  own part and honestly says the rest is a different asset type, rather
  than either hallucinating or leaving the user stuck. A dedicated
  comparison agent (reusing existing tools across domains) would close
  this; still on the backlog.
- No watchlist/portfolio tracking, no proactive price alerts — both would
  need real new infrastructure (a persistent per-user holdings table, and
  for alerts specifically a scheduler + a delivery channel), explicitly
  deferred by choice rather than oversight.
- tool_call_log doesn't durably record which SPECIFIC tool/agent handled
  each chat turn — invocation_log's agent_id is always the top-level
  chatbot root (market_intelligence), so admin usage-by-agent breakdowns
  for chat traffic will show one row, not a per-specialist split. The
  self-service /chat/usage "by specialist" view inherits this same
  limitation. Fixing it would mean threading tool/agent identity into
  ToolCallLog writes from _run_turn/_stream_turn, not yet done.
- Competitors, ESG scores, credit ratings — no free/no-key data source
  exists; agents are instructed to say so rather than fabricate.
- Multi-tenancy (workspace_id) exists in the schema everywhere but isn't
  fully enforced as a hard boundary yet.
- Real OpenTelemetry export / Langfuse integration — IMPLEMENTED (see
  section 18, DEBUG CONSOLE / DISTRIBUTED TRACING). Off by default
  (OTEL_ENABLED=false); the Debug Console's DB-reconstructed waterfall works
  with it off, real span-level tracing needs a reachable OTLP backend.

- Backend dev server: `uvicorn app.main:app --reload` from backend/, port

- If GEMINI_API_KEY-dependent calls start failing with "No API key was

  Settings singleton.
- Debug Console tracing: `docker compose up -d jaeger` (repo root), then set
  OTEL_ENABLED=true in backend/.env and restart. Jaeger UI/Query API on
  :16686, OTLP gRPC receiver on :4317. Off by default — the backend runs
  fine, and the Debug Console still works (DB-reconstructed mode), without
  Jaeger running at all.

17. THE DEVELOPER ROLE & PUBLISH-APPROVAL WORKFLOW
==============================================================================

17.1 WHY A FOURTH ROLE
------------------------------------------------------------------------------
The original three roles (admin/viewer/chat_user) had no "can build agents
but isn't fully trusted to ship them" tier — every config-writing endpoint
was admin-only. `developer` fills that gap: enough access to be productive
building/testing agent trees end-to-end, without a single compromised or
careless developer account being able to change what a real end user's
/chat session actually talks to.

17.2 TWO INDEPENDENT APPROVAL GATES
------------------------------------------------------------------------------
There are deliberately TWO separate approval checkpoints, not one:

  GATE 1 — ACCOUNT APPROVAL (existing pending/approved/rejected on `users`)
    A developer self-registers at POST /auth/register with role=developer
    (SELF_SERVE_ROLES = {"chat_user", "developer"} — admin/viewer stay
    admin-only-creatable via POST /auth/users, never self-serve). The
    account sits `pending` until an admin approves it at
    POST /auth/{id}/approve, exactly like a chat_user signup today. This is
    a one-time "let this person in the door at all" decision.

  GATE 2 — PER-PUBLISH APPROVAL (new: `agent_publish_requests` table)
    Once approved, a developer has real write access — create agents,
    attach tools/skills/sub-agents, run them in the Playground — but calling
    POST /agents/{id}/publish as a developer NEVER makes the agent live.
    Instead it freezes the current live config into a new
    AgentPublishRequest row (status=pending) and stops. An admin reviews it
    at GET /agents/publish-requests and either:
      POST /agents/publish-requests/{id}/approve — publishes EXACTLY the
        snapshot that was frozen at request time (not whatever the live
        draft has drifted to since — what was reviewed is what ships),
        creating a real AgentVersion row via the same _publish_now() admin
        publishing already used, and stamping published_version back onto
        the request.
      POST /agents/publish-requests/{id}/reject — with an optional
        review_note the developer can see at GET /agents/publish-requests/mine.
    An agent can have at most one PENDING request at a time (a second
    publish attempt while one is outstanding is rejected with 409) — a
    developer iterating further just has to wait for the first decision or
    keep working in draft; there's no way to spam the queue.

    Admin publishing is completely unaffected: an admin's POST
    /agents/{id}/publish still goes live immediately, exactly as before
    section 7.8 describes. The branch is entirely on `principal.role`.

17.3 OWNERSHIP SCOPING
------------------------------------------------------------------------------
A developer can only CREATE new agents (created_by is force-set to their own
identity server-side, never trusted from the request payload — see
config_api.agents.create_agent) and can only MODIFY (update/archive/attach-
detach tools|skills|sub-agents/publish) agents where agents.created_by
matches their own identity (config_api.agents._require_can_modify). This
stops one developer from silently rewriting another developer's — or an
admin's — agent tree, while still letting every developer freely READ the
whole workspace's agents (list/get/versions are admin+viewer+developer) so
they can attach an existing shared Tool/Skill or wire in someone else's
published agent as a sub-agent.

Tools remain strictly admin-only to create/edit/delete (http_tool/sql_tool/
mcp_tool configs are a real security boundary — arbitrary URLs, SQL text,
subprocess commands). Skills (pure instruction-text fragments, no execution
surface) are open to developer create/update/delete, same as agents.
rollback_agent is deliberately admin-only even for agents a developer owns —
rollback republishes immediately with no review step, which would be a
straight bypass of gate 2 above.

17.4 PERMISSION MATRIX
------------------------------------------------------------------------------
                          admin   viewer   developer   chat_user
  Agents: create/edit       Y       -      Y (own)        -
  Agents: read (list/get)   Y       Y         Y            -
  Tools/Skills: create       Y       -    skills only       -
  Playground run            Y       -         Y            -
  Publish (own agent)     immediate  -    -> review queue   -
  Publish approve/reject    Y       -         -             -
  Rollback                  Y       -         -             -
  Chat (/chat/*)             Y       -         Y            Y
  Own usage (/chat/usage)    Y       -         Y            Y
  Admin dashboards (RO)      Y       Y         -             -
  Debug Console               Y       Y     Y (scoped*)      -
  Users admin                 Y       -         -             -

  * developer's Debug Console view is scoped to traces for agents they
    created OR invocations they personally triggered (own chat/playground
    activity) — see section 18.4.

17.5 FRONTEND SURFACE
------------------------------------------------------------------------------
A developer signs into the SAME admin shell as admin/viewer (not a separate
app) via LoginPage's email/password tab (POST /auth/login — previously this
tab only existed for the /chat surface; the admin shell's login now accepts
either the static break-glass token OR any approved named account, admin/
viewer/developer alike). The sidebar (Layout.tsx) renders role-appropriate
navigation: a developer sees Agents/Tools(read-only)/Skills/Playground/Chat/
Debug Console/"My publish requests", never Users or the full admin
dashboards. An admin additionally sees a "Publish requests" queue with an
unread-count badge for pending reviews.

18. DEBUG CONSOLE / DISTRIBUTED TRACING ARCHITECTURE
==============================================================================

18.1 THE PROBLEM
------------------------------------------------------------------------------
A single /chat turn can fan out across a whole tree: orchestrator receives
the message, transfers to a specialist, that specialist calls two or three
tools (each an MCP stdio subprocess round-trip), maybe transfers back and
across to a second specialist. Section 15's original limitation list called
this out explicitly: no durable per-agent/per-tool attribution existed for a
chat turn, only an aggregate InvocationLog row for the top-level chatbot
root. Debugging "why did this take 14 seconds" or "which specialist actually
answered this" meant reading application logs by hand. This section closes
that gap with an actual waterfall/timeline view, backed by real distributed
tracing infrastructure that plugs into whatever the surrounding org already
runs (or nothing at all, in DB-only mode).

18.3 TWO OPERATING MODES, ONE UI
------------------------------------------------------------------------------
Every span-creation call site uses `app.observability.tracing.get_tracer()`,
which returns:
  - a real tracer (exporting via OTLP/gRPC to OTEL_EXPORTER_OTLP_ENDPOINT)
    once `setup_tracing(app)` has installed a TracerProvider — gated on
    settings.otel_enabled, checked once at process startup;
  - otherwise the OpenTelemetry API's built-in no-op tracer, which has the
    IDENTICAL method signatures but every span is a free no-op with an
    invalid (all-zero) SpanContext.
Call sites never branch on whether tracing is on — `get_tracer()` makes that
distinction disappear. The one place that DOES check is right after span
creation: `span_context.is_valid` gates whether an otel_trace_id/otel_span_id
gets persisted at all, so a disabled-tracing run correctly stores NULL
rather than a misleading all-zero id.

GET /debug/traces/{invocation_id} tries the live path first — if
otel_enabled and the row has a non-null otel_trace_id, it calls the Jaeger
Query API (`GET {JAEGER_QUERY_URL}/api/traces/{otel_trace_id}`) with a short
(2.5s) timeout, reshapes Jaeger's span format (references[].refType==
CHILD_OF for parent linkage, startTime/duration in microseconds, tags[] for
agent.name/otel.status_code) into the same SpanNode shape the frontend
renders either way, and returns spans_source="jaeger" — real wall-clock
timing, works even for concurrent/overlapping tool calls, and ships a
jaeger_trace_url deep link ("Open in Jaeger" in the UI) for the full
Jaeger UI (search across traces, service maps, tag-based queries, latency
histograms — anything the Debug Console doesn't itself replicate).

If Jaeger isn't reachable, isn't enabled, or the trace hasn't landed there
yet (network unreliability, or debugging a moment right after the call
before the batch exporter flushed), the fetch fails silently and
GET /debug/traces/{id} instead reconstructs a waterfall directly from
Postgres (spans_source="reconstructed"): one root span from the InvocationLog
row itself, one child span per ToolCallLog row ordered by call_index (NOT
created_at — every one of an invocation's ToolCallLog rows is written in a
single fire-and-forget batch AFTER the run finishes, so created_at alone
can't recover call order), laid out sequentially by cumulative latency_ms.
This is an approximation (true parallel tool calls would render as
back-to-back rather than overlapping), clearly labeled as such via
spans_source, but requires ZERO extra infrastructure — it works on a bare
`uvicorn app.main:app` with nothing else running.

18.4 SECURITY / MULTI-TENANCY
------------------------------------------------------------------------------
Every debug_api query is filtered by `InvocationLog.workspace_id ==
principal.workspace_id` first (same tenant boundary as every other
dashboards_api endpoint). On top of that, a developer's view is further
scoped (config_api parity with 17.3's ownership rule) to traces where
EITHER the invoked agent's `created_by` matches their own identity OR the
invocation's `invoked_by` matches their own user id (own chat/playground
activity on someone else's published agent) — enforced both in the list
query's WHERE clause and, defense-in-depth, again in
`_get_invocation_scoped` for the single-trace detail endpoint (a 403, not a
404, so a developer gets an honest "not yours" rather than a misleading
"doesn't exist"). admin and viewer see the full workspace, matching their
existing access to /dashboards/audit/invocations.

18.5 PLUGGING INTO EXISTING INFRA
------------------------------------------------------------------------------
Nothing here is Jaeger-specific except the Query-API convenience integration
in 18.3 — the actual instrumentation exports standard OTLP/gRPC, so pointing
OTEL_EXPORTER_OTLP_ENDPOINT at a different collector (Grafana Tempo,
Langfuse's OTel ingestion endpoint, an OpenTelemetry Collector fronting
Honeycomb/Datadog/New Relic/whatever the org already pays for, etc.) works
with zero code changes — spans show up there exactly as they would in
Jaeger. Only the "Debug Console reads spans back for its own UI" half (18.3)
is Jaeger-Query-API-shaped today; swapping that reader for a different
backend's query API (or removing it and relying purely on the "Open in
Jaeger"-style deep link plus the always-available DB-reconstructed fallback)
is a contained change inside `_fetch_jaeger_spans()` and the
`jaeger_query_url`/`jaeger_ui_url` settings — nothing about how spans are
CREATED needs to change.

18.6 WHAT THIS DOES NOT DO (BY CHOICE, THIS BUILD)

- MCP servers run as separate stdio subprocesses (mcp_servers/*.py, section
  9.2) and are NOT themselves OTel-instrumented — a tool-call span in the
  backend process covers "how long the MCP round-trip took" as one black
  box, not a further breakdown of what happened inside mfapi.in/Yahoo
  Finance/etc. calls. HTTPXClientInstrumentor DOES cover outbound httpx
  calls made directly BY the backend process itself (e.g. the Jaeger Query
  API fetch), just not calls made from inside a spawned MCP subprocess.
- No sampling/retention policy configuration is exposed — that's entirely
  the trace backend's own concern (Jaeger's own retention/storage config,
  or whatever a hosted OTel vendor offers) once OTEL_EXPORTER_OTLP_ENDPOINT
  points at it.
- Tool call input/output are NOT currently attached as span attributes (only
  name/agent/latency/status) to avoid putting potentially large or sensitive
  payloads into a trace backend by default; they ARE available in the
  DB-reconstructed root span's message/response_text and in the existing
  /dashboards/audit/invocations transcript. Worth revisiting behind an
  explicit opt-in if full request/response capture in traces is wanted.
