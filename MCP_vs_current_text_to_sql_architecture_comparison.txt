EÄRENDIL — ARCHITECTURE COMPARISON
Current text-to-SQL structure (data_query_tool) vs. a proposed MCP-based
database interface layer, with SCIL validators as middleware
==============================================================================

CONTEXT / THE PROPOSAL BEING EVALUATED
------------------------------------------------------------------------------
Combine options 3 + 4: use MCP as the standardized database interface layer,
and Agent Forge's declarative config as the agent definition. SCIL
validators sit as middleware. New databases become a config row + an MCP
server endpoint — zero agent code. This also future-proofs the platform if
the wider enterprise GenAI platform standardizes on MCP.

This document grounds that proposal against what's actually implemented
today (verified against the real code, not assumed), identifies one real
gap in the proposal as stated, and lays out a recommended hybrid path.

==============================================================================
PART 1 — WHAT "CURRENT" ACTUALLY IS, PRECISELY
==============================================================================
`data_query_tool.py` (backend/app/tool_registry/data_query_tool.py) is NOT a
generic MCP-style tool — it's a `ConfigDrivenTool` subclass that does three
things in one Python function, in-process, in this order:

  1. Opens a pooled `pymysql` connection directly (credentials from
     os.environ, connection picked by the data_entities row's `connection`
     config)
  2. Validates the LLM's SQL via a real `sqlglot` AST — single SELECT, no
     DML/DDL, exactly one table
  3. Mechanically ANDs the caller's row-level-security predicate onto the
     parsed AST (`build_policy_condition`) — populated from
     `policy_engine.apply_policy`, itself invoked by an ADK
     `before_tool_callback` (app/agent_runtime/builder.py:36) that fires
     before ANY tool call the agent makes, once per invocation

That `before_tool_callback` is the load-bearing piece for security — it's a
generic ADK-level hook registered once on the built agent
(builder.py:161,231), not something baked specifically into
`data_query_tool`. This matters for Part 2.

==============================================================================
PART 2 — WHAT THE MCP PROPOSAL GETS RIGHT, AND ONE GAP IT DOESN'T MENTION
==============================================================================
Checked how `mcp_tool` actually works today (app/tool_registry/mcp_tool.py):
it binds the agent's LLM DIRECTLY to an `McpToolset` — the model calls the
MCP server's tool straight, with no `ConfigDrivenTool` wrapper in between.

That's a real gap in the "config row + MCP endpoint = zero code" framing:
as built today, an mcp_tool-typed database tool would BYPASS BOTH the
sqlglot guardrail AND the RLS predicate injection entirely, because those
live inside `DataQueryTool.run_async`, not in anything an MCP call passes
through.

The good news: `before_tool_callback` is an ADK-level hook that fires for
EVERY tool call regardless of tool type — including McpToolset-backed ones.
So closing this gap doesn't require a rewrite; it requires extending that
one existing callback to also intercept MCP-typed tool args when a
`policy_id` is present, the same way it already does for `data_query_tool`.

DIMENSION: Reusability outside Agent Forge
  Current:  None — data_query_tool is Agent Forge-specific Python.
  Proposed: High — any MCP-compliant client (a different internal GenAI
            app, Claude Desktop, another agent framework) can reuse the

DIMENSION: Ops surface
  Current:  One process, one set of DB credentials to secure (Agent Forge's

  Proposed: N additional deployable processes, each with its own credential
            handling — more failure-isolated, but more to monitor and
            secure per domain.

  Current:  In-process DB round-trip (milliseconds).
  Proposed: +1 network/subprocess hop per query — likely negligible next to

  Current:  Already true today — validate_output runs in

            of which tool answered.

            buys you.

==============================================================================
PART 4 — RECOMMENDED ALTERNATIVE: MCP AS A PURE EXECUTION/TRANSPORT LAYER,
NOT A QUERY-AUTHORING LAYER
==============================================================================
Don't hand the LLM a raw MCP database tool and let it write + send SQL
end-to-end — that's where the guardrail gap above comes from, and it means
re-litigating "is this MCP server's SQL safe" once per server, forever.
Instead, keep the split that already exists today and just swap what's on
the far side of it:

  1. KEEP data_query_tool's validation and policy-injection code exactly
     as-is — it's already generic, already tested, already the one place
     RLS is enforced. Don't move this logic into N different MCP servers
     of varying trustworthiness.

  2. Replace ONLY the execution step: instead of data_query_tool opening
     its own pymysql connection, it calls a thin, generic `execute_sql`
     MCP tool — one per DB engine (Postgres, MySQL, Mongo), each doing
     nothing but "run this already-validated, already-scoped query string,
     return rows."

  3. The onboarding wizard's "connection" step offers two paths: a native
     connector (today's behavior, unchanged) OR point at any external MCP
     server that already speaks that DB engine — best of both, not a
     forced migration.

  4. Extend before_tool_callback to also merge apply_policy's result into
     MCP-typed tool call args when a tool config carries a policy_id —
     same mechanism already in production, one more tool type recognized.

WHY THIS IS THE RIGHT SHAPE:
This gets the enterprise-standardization and reusability benefit (the
actual wire protocol becomes MCP, future-proofed against enterprise-wide
MCP standardization) WITHOUT moving security-critical logic outside a
codebase this team controls and can audit — which is the real risk hiding
in "new databases become a config row + MCP endpoint, zero agent code":
that phrase is only safe if the VALIDATION stays centralized, and as the
platform is built today, routing raw DB access through mcp_tool would
silently not preserve that.
